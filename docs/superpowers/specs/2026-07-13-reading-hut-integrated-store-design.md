# 阅读小屋家具/装饰与星星兑换整合设计

日期：2026-07-13  
状态：已完成讨论，等待书面规格复核  
目标版本：1.0  
目标用户：小学 1–6 年级

## 1. 背景与目标

阅读小屋当前在房间底部提供四个入口：家具、装饰、图鉴和星星商店。独立商店使孩子需要在“查看商品、兑换商品、返回房间、再次打开背包、摆放商品”之间往返。

1.0 将商店能力合并到家具和装饰面板，形成以下核心循环：

> 完成每日学习计划 → 获得星星 → 兑换家具或装饰 → 返回面板 → 选择已拥有物品进行摆放

设计目标：

- 缩短从获得星星到兑换物品的路径。
- 让孩子清楚区分未拥有、已拥有和使用中的物品。
- 让余额不足自然指向每日学习计划，而不是形成死路。
- 保留图鉴的收集与成就价值，避免其与兑换功能重复。

## 2. 1.0 范围与非目标

### 2.1 范围

- 家具与装饰面板同时展示已拥有和未拥有物品。
- 未拥有物品在卡片右上角显示星星价格。
- 点击未拥有物品进入详情与兑换确认。
- 兑换成功后更新余额、物品状态和图鉴进度。
- 余额不足时可跳转至现有每日学习计划页面。
- 已拥有物品可从家具或装饰面板进入摆放流程。

### 2.2 非目标

- 不保留独立星星商店入口或页面。
- 不使用等级、章节或任务物品锁。
- 不支持同一种物品拥有多份。
- 不使用限购次数或库存逻辑。
- 不支持兑换成功后自动摆放。
- 不在图鉴内完成兑换。
- 不在 1.0 添加“全部/已拥有/可兑换”等筛选器。
- 不涉及真实货币或付费购买。

## 3. 信息架构与命名

房间底部从四个入口缩减为三个：

1. 家具
2. 装饰
3. 图鉴

移除“星星商店”。“家具背包”和“装饰背包”统一改称“家具”和“装饰”。原因是面板会展示未拥有物品，继续使用“背包”会与儿童对“背包只包含我的物品”的直觉冲突。

各入口职责：

- **家具/装饰**：浏览、兑换、查看拥有状态、进入摆放。
- **图鉴**：展示完整收藏范围、分类进度和总进度。

房间顶部继续展示当前星星余额。

## 4. 物品状态与卡片表达

1.0 不显示锁头。物品只有以下用户可见状态：

| 状态 | 卡片表达 | 点击行为 |
| --- | --- | --- |
| 未拥有、余额充足 | 正常彩色物品；右上角黄色“星星图标 + 价格” | 打开物品详情与兑换确认 |
| 未拥有、余额不足 | 物品仍可见；价格弱化；文字提示星星不足 | 打开余额不足弹窗 |
| 已拥有、未摆放 | 绿色“已拥有”；不显示价格 | 进入摆放流程 |
| 使用中 | 强化边框；绿色“使用中”；不显示价格 | 不重复摆放；提示“已在房间中” |

约束：

- 状态不能只靠颜色表达，必须同时使用文字、形状或边框。
- 同一张卡片不能同时展示价格和“已拥有/使用中”。
- 兑换成功后，卡片短暂显示 `NEW`，随后保留“已拥有”。
- 每种物品最多拥有一份；已拥有物品不可再次兑换。

## 5. 兑换流程

### 5.1 余额充足

1. 用户点击未拥有物品。
2. 系统打开详情/确认弹窗，展示物品图片、名称、价格、当前余额和兑换后余额。
3. 弹窗提供“再想想”和“确认兑换”。
4. 用户点击“确认兑换”后，按钮立即禁用，防止重复提交。
5. 系统再次校验物品尚未拥有且余额足够。
6. 成功后一次性完成扣减余额与写入拥有状态。
7. 弹窗关闭，播放简短获得反馈，并显示“兑换成功，已添加到家具”或“兑换成功，已添加到装饰”。
8. 当前面板中的卡片变为“已拥有”，星星余额同步更新，图鉴进度同步增加。
9. 用户留在当前面板；系统不自动摆放该物品。

### 5.2 余额不足

余额不足弹窗展示：

- 物品名称和价格。
- 当前余额。
- “还差 X 颗星”。
- “完成每日学习计划可以获得星星”。
- “暂不兑换”和“去完成计划”。

“去完成计划”连接现有计划首页中的每日阅读计划区域。该跳转属于 1.0 正式交互要求。

### 5.3 失败与并发保护

- 点击确认后禁用按钮，避免连续点击造成重复扣费。
- 确认时重新校验余额和拥有状态。
- 如果余额在弹窗打开后发生变化并变得不足，切换为余额不足反馈，不扣费。
- 如果兑换失败或网络异常，不扣星星、不改变物品状态，并提示“兑换没有成功，请再试一次”。
- 失败后恢复按钮，允许重试。
- 错误文案和音效不使用惩罚或责备语气。

## 6. 图鉴联动

- 图鉴展示全部家具和装饰，但不显示价格，也不提供兑换按钮。
- 未收集物品使用灰色剪影表示“尚未拥有”，不显示锁头。
- 已收集物品恢复彩色并显示绿色对勾。
- 移除图鉴卡片右上角用于表示“已收集”的黄色星星，避免星星同时表达货币和收藏状态。
- 兑换成功后同步更新对应图鉴卡片、分类收集数和总收集数。
- 点击图鉴中的未收集物品时，可展示物品名称与提示“前往小屋的家具/装饰中兑换”，但不能直接购买。

符号规则：

- 星星只表达货币。
- 绿色对勾只表达已收集。
- 灰色剪影只表达未拥有。

## 7. 状态模型与数据流

### 7.1 数据结构

物品定义包含：

- `id`
- `category`：`furniture` 或 `decoration`
- `name`
- `image`
- `price`

用户状态包含：

- `starBalance`
- 每个物品的 `owned` 布尔值
- 每个物品的 `placed` 布尔值；1.0 不记录自由摆放坐标
- 新获得物品的临时 `new` 展示状态

`affordable` 不单独持久化，由 `!owned && starBalance >= price` 实时计算。分类收集数与总收集数由 `owned` 状态实时计算，避免家具/装饰面板和图鉴产生两套进度数据。

### 7.2 兑换事务

兑换操作必须以同一事务完成以下两项：

1. 扣减星星余额。
2. 将物品标记为已拥有。

任一步失败都不得保留部分结果。成功后，家具/装饰卡片、顶部余额和图鉴都读取同一份更新后的状态。

## 8. 组件边界

- **房间底部导航**：只负责打开家具、装饰或图鉴。
- **物品面板**：按类别读取并展示全部物品。
- **物品卡片**：只根据输入状态渲染价格、已拥有或使用中，并发出点击意图。
- **兑换弹窗**：展示详情，处理确认、取消、余额不足和失败反馈。
- **兑换服务/状态层**：执行余额和拥有状态的原子更新。
- **图鉴**：读取拥有状态并计算分类与总进度。
- **计划跳转**：将余额不足用户送至现有每日计划入口。

组件之间共享物品定义、钱包和拥有状态，不各自维护副本。

## 9. 小学 1–6 年级适配

- 图标同时配文字，不能假定儿童只看图就能理解。
- 关键状态不只依赖颜色。
- 主要点击区域不小于约 44 × 44 px。
- 弹窗一次只要求做一个决定，使用短句和具体数字。
- 兑换前必须确认，兑换后不自动摆放。
- 星星不足时明确给出下一步，避免只显示失败。
- 可用性测试分为 1–3 年级和 4–6 年级两组，不能用高年级测试结果代表全部小学生。

## 10. 验收与测试

功能验收覆盖：

1. 房间底部只显示家具、装饰和图鉴三个入口。
2. 家具与装饰分别展示对应类别的全部物品。
3. 余额大于或等于价格时可兑换；余额恰好等于价格时兑换后余额为零。
4. 余额不足时不扣费，并能跳转至每日计划。
5. 连续点击确认不会重复扣费或重复写入拥有状态。
6. 兑换成功后余额、物品卡片、分类进度和总图鉴进度同步更新。
7. 兑换成功后不自动进入摆放流程。
8. 已拥有物品不能重复兑换。
9. 兑换失败时余额与拥有状态保持不变。
10. 刷新或重新进入后余额和拥有状态仍然存在。
11. 图鉴中不出现价格、购买按钮、等级锁或表示收藏状态的黄色星星。
12. 家具与装饰的状态与交互规则一致。

儿童可用性测试任务：

1. 识别当前拥有多少星星。
2. 找到一个买得起的物品并完成兑换。
3. 找到兑换后的物品，理解它已经属于自己但尚未摆放。
4. 尝试兑换一个买不起的物品，并说明接下来如何获得星星。
5. 进入图鉴并判断哪些物品已经收集。

重点观察是否出现以下误解：

- 把图鉴中的状态符号理解为价格。
- 认为灰色物品永远不能获得。
- 认为兑换成功后物品已经自动摆进房间。
- 不知道余额不足后应去完成学习计划。
- 把“家具/装饰”面板误认为只包含已拥有物品。

## 11. 最终决策摘要

- 采用“中性名称 + 已拥有/未拥有混合列表”。
- 移除独立商店和“背包”命名。
- 1.0 不使用筛选、等级锁、多份库存或限购。
- 未拥有物品以右上角星星价格表达兑换入口。
- 兑换需要确认，成功后留在面板，不自动摆放。
- 余额不足时引导至每日学习计划。
- 图鉴只负责收集进度，不负责兑换。
- 目标用户限定为小学 1–6 年级，并分年龄段测试。
